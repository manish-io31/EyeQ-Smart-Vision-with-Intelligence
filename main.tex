\Large{\paragraph\\
Surveillance systems increasingly require timely interpretation of video streams rather than passive recording alone. This project presents EyeQ -- Smart Vision with Intelligence, a web-based intelligent surveillance platform that performs near real-time object detection on live camera input and provides operator awareness through a centralized dashboard and alert history. The system captures video from common sources such as webcams and RTSP streams, transmits compressed frames to a Python Flask backend, and applies a hierarchical detection strategy: AWS Rekognition when cloud credentials are available, otherwise YOLO (Ultralytics) for local inference, with a demo mode to preserve usability during setup or partial failures. Detections are normalized into a unified representation with labels, confidence scores, and bounding boxes for consistent visualization.
EyeQ integrates policy-driven alerting using configurable confidence thresholds and threat-label matching, together with per-label cooldown to reduce notification spam. Notifications can be dispatched through SMS (Twilio), Telegram, and email (SMTP), while events are persisted in a relational database (SQLite by default, with PostgreSQL as an optional deployment choice) for auditing and analytics. A web dashboard summarizes detection and alert activity using charts and recent event listings. The proposed solution demonstrates a practical end-to-end workflow from frame acquisition to automated alerting, suitable for institutional monitoring scenarios subject to responsible deployment and governance.

\chapter{INTRODUCTION}
\pagenumbering{arabic}
\section{Introduction}
\hspace{0.5cm}
In recent years, video surveillance has become a routine safeguard in campuses, workplaces, and public facilities. Conventional deployments often depend on continuous human monitoring of multiple camera feeds, which is cognitively demanding and difficult to sustain over long shifts. Even when recordings are available for later review, the process is reactive: incidents may be noticed only after damage or loss has occurred. These limitations motivate the integration of intelligent components that can interpret visual data quickly and highlight important events to operators.

Advances in Artificial Intelligence (AI) and Computer Vision (CV) have made it practical to assist human security workflows through automated perception. Modern detectors can localize objects in images and video frames with strong speed--accuracy trade-offs, enabling near real-time feedback in interactive applications. At the same time, web technologies allow such capabilities to be delivered through browser-based dashboards, improving accessibility for administrators and reducing the need for specialized workstation software for routine monitoring tasks.

This project presents EyeQ -- Smart Vision with Intelligence, an end-to-end intelligent surveillance platform designed to monitor live visual input and support timely situational awareness. EyeQ captures frames from common sources such as webcams and RTSP streams, transmits them to a Python Flask backend, and performs detection using a practical fallback strategy: AWS Rekognition when cloud credentials are configured, otherwise YOLO inference locally, and finally a controlled demo mode to preserve system usability during setup or partial failures. Detected events can be recorded for auditing, while alert-worthy findings can trigger notifications through SMS (Twilio), Telegram, and email, subject to configurable thresholds and cooldown rules to reduce redundant alerts.

The system also provides a web dashboard with summary statistics and recent activity, enabling operators to review trends without manually scanning raw footage. Overall, EyeQ aims to reduce dependence on uninterrupted manual screening, improve traceability through structured logs, and accelerate response by combining automated detection with multi-channel alerting in a unified architecture.

\section{Background}
\hspace{0.4cm}
Traditional CCTV systems primarily record footage for post-incident investigation. While recording is necessary, it does not automatically translate into timely prevention or rapid response. Security teams must either watch screens continuously or spend significant time searching archives, which becomes impractical as camera counts and retention durations increase.

Recent progress in machine learning has enabled robust object detection in unconstrained scenes. Methods such as YOLO (You Only Look Once) support fast inference suitable for interactive applications, while cloud vision APIs can reduce local compute requirements when network connectivity and operational policies permit their use. Integrating such technologies with a structured software workflow---authentication, APIs, persistence, and alerting---moves surveillance from passive capture toward assisted monitoring. EyeQ follows this direction by combining detection backends with a web application layer that is suitable for institutional demonstration and iterative improvement.
\section{Objective}
The primary objective of this project is to design and implement EyeQ -- Smart Vision with Intelligence, a web-based intelligent surveillance system that processes live video input and presents detection outcomes through a unified operator interface. The system targets reliable end-to-end execution: authenticated access, frame ingestion from webcams and RTSP sources, detection and visualization with bounding boxes, persistence of detection/alert records, and configurable notification delivery. A further objective is to maintain operational flexibility through a hierarchical detection pipeline that can leverage cloud inference when available and local inference when required, rather than forcing a single deployment assumption.

A secondary objective is to improve practical security operations through timeliness and traceability. Timeliness is addressed by near real-time detection loops and alert channels that can inform responsible stakeholders quickly when configured threat conditions occur. Traceability is addressed by storing structured event history in a database (SQLite by default, with PostgreSQL supported for scalable deployments) and by providing dashboard analytics for summarized monitoring. The project also emphasizes engineering completeness---modular routes/services, environment-based configuration, and safeguards such as confidence thresholds and per-label cooldown---so the solution remains maintainable and extensible for future enhancements such as stricter role policies, asynchronous notification queues, or deployment hardening for production environments.

\section{Problem Statement}
Conventional monitoring workflows are limited by human scalability. As the number of cameras grows, continuous manual observation becomes expensive, inconsistent, and prone to missed events during fatigue or distraction. Recording-only systems add storage and retrieval burden: finding a specific incident may require lengthy manual review unless meaningful metadata (object classes, timestamps, confidence, alert flags) is generated automatically at capture time. In addition, many ad-hoc scripts or standalone detection demos lack integrated user management, persistent history, and practical alerting---which reduces their usefulness as deployable surveillance tools.

EyeQ addresses these gaps by providing an integrated platform that combines computer vision with a full application stack. The system reduces exclusive reliance on continuous human watching by automatically highlighting detections and generating alert events when policy conditions are satisfied. It also reduces reactive archive search by maintaining structured logs and dashboard summaries for operational review. Practically, the work must balance competing constraints---accuracy versus false alarms, latency versus model complexity, cloud dependence versus local compute---through configurable thresholds and backend selection. The intended outcome is not to replace human judgment, but to assist operators with faster awareness, better documentation of events, and scalable monitoring suitable for institutional environments, implemented with responsible use and access control considerations.

\chapter{LITERATURE REVIEW}

\section*{Overview}
The field has transitioned from classical computer vision pipelines---such as background subtraction, frame differencing, optical-flow cues, and Haar-cascade detectors---to learning-based representations that generalize better under clutter, illumination change, and viewpoint variation. While classical methods remain lightweight and interpretable, they frequently degrade in unconstrained scenes due to dynamic backgrounds, shadows, and repetitive textures. Consequently, research momentum shifted toward deep learning, where convolutional architectures learn hierarchical features directly from data.

At the same time, deployment constraints created two dominant directions: (i) edge/local inference using fast detectors (e.g., YOLO-family models) to minimize latency and recurring cloud cost, and (ii) managed cloud vision APIs (e.g., AWS Rekognition) that reduce local hardware requirements and simplify scaling, at the expense of network dependency, latency variability, and operational cost. Practical surveillance systems therefore increasingly combine algorithm selection, thresholding/policy design, and application integration (authentication, storage, dashboards, alerting), rather than focusing on detection accuracy alone.

\section{Existing System}
\textbf{Manual verification and operational limits.} Many legacy deployments rely on continuous human monitoring. This approach does not scale with camera count or retention duration: operator fatigue increases missed events, and incident discovery may occur only after a complaint or audit request. Even when video is recorded, reviewing archives without structured metadata remains slow and labor-intensive.

\textbf{Scalability and data growth.} As resolution, frame rate, and camera deployments increase, raw video volume grows super-linearly relative to staffing. Organizations therefore need automated indexing signals (labels, timestamps, confidence, camera id) to narrow searches and prioritize review.

\textbf{Classical automated methods.} Early automated assistance commonly used motion detection and background modeling. These methods can work in controlled scenes but often produce frequent false positives outdoors (wind, rain, lighting flicker). Haar-like cascades and hand-crafted features improved specific tasks (e.g., frontal face detection) but were brittle across pose, occlusion, and distance.

\textbf{Performance and environment mismatch.} Traditional pipelines may perform well in laboratory setups yet fail in crowded, occluded, or low-light environments. Moreover, detection is only one component: without integrated workflows (roles, audit logs, alert policies), technical accuracy does not translate into dependable operations.

\section{Related Work}
\textbf{Deep learning for detection and understanding.} Convolutional neural networks became the standard backbone for object detection and recognition. Single-stage detectors such as YOLO and SSD emphasize speed--accuracy trade-offs suitable for near real-time use, while two-stage detectors often prioritize accuracy at higher compute cost. These advances enabled robust object localization in diverse scenes, supporting surveillance analytics beyond simple motion triggers.

\textbf{Human-centric and behavioral cues.} Beyond object boxes, research explored face detection/attributes, pose estimation, and activity cues. Such cues can improve situational interpretation, though they also raise privacy and governance considerations in deployment.

\textbf{Cloud vision services and managed APIs.} Cloud platforms provide pretrained models and standardized interfaces that reduce engineering time for teams without large ML infrastructure. For example, AWS Rekognition supports label/scene cues, face analysis, text detection, and moderation signals depending on configuration. These services are attractive for rapid prototyping and elastic workloads, while local models remain relevant for offline operation, predictable latency, or cost control.

\section{Research Gap}
Despite progress, several gaps remain relevant to integrated campus-scale surveillance platforms:

\textbf{Robustness under live-stream variability.} Real-time feeds differ from benchmark datasets due to motion blur, compression artifacts, low light, and occlusions. Maintaining stable precision/recall across cameras and scenes remains challenging; systems must often tune thresholds and policies rather than relying on a single fixed model score.

\textbf{Metadata management and operational retrieval.} Continuous monitoring generates large volumes of events. Many solutions emphasize detection but provide limited structured logging, filtering, and analytics for long-running operations. Efficient indexing and summarization (hourly trends, top labels, alert timelines) are essential for usability.

\textbf{Integrated visualization and decision support.} Analysis modules are insufficient without operator interfaces that reduce cognitive load: live overlays, alert history, role-based access, and summarized metrics. A research gap exists between ``model accuracy demos'' and deployable workflows with auditing and notification discipline.

\textbf{False alarms and policy control.} Occlusion, overlapping objects, and scene complexity can increase false positives. Practical systems need cooldown strategies, confidence gating, threat-label policies, and human-in-the-loop review---areas often underrepresented in pure model papers.

\textbf{Deployment diversity (cloud vs edge).} Many publications focus on a single deployment mode. There is practical value in architectures that support \textbf{fallback strategies} (cloud API when available, local inference otherwise) and maintain continuity for demonstration, training, and constrained environments---a gap EyeQ targets at the engineering-integration level.

\section*{Summary}
The literature supports a clear direction: combine strong detection backends with robust application integration, responsible alert policy, and operational analytics. The proposed EyeQ platform aligns with this direction by unifying live detection, persistence, dashboard reporting, and multi-channel alerting under a modular Flask-based architecture.

\section{Existing System}

In many institutions and organizations, surveillance relies on traditional CCTV systems where cameras continuously record footage and security personnel manually monitor screens. This approach depends heavily on human attention, especially when handling multiple video feeds over long durations. As a result, operators may miss critical events due to fatigue, distractions, or delayed response. Additionally, conventional systems only store video data without providing intelligent analysis, making it difficult to detect suspicious activities such as unauthorized access or unsafe behavior in real time.

Another major limitation of existing systems is the lack of automated alerting and structured data management. Most systems do not provide instant notifications through SMS, Telegram, or email, which delays response time. Searching recorded footage is time-consuming due to the absence of metadata such as timestamps and object labels. Therefore, traditional surveillance systems are inefficient, reactive, and unable to meet modern security requirements.

\section{Algorithm \& Pseudo Code}

\subsection{Purpose}
This section describes the complete processing pipeline of the EyeQ system, from capturing video frames to generating alerts and storing events.

\subsection{Input Acquisition}
The browser captures frames from webcam, RTSP streams, or local video files. These frames are compressed (e.g., JPEG format) and encoded in base64 before being sent to the backend via the POST \texttt{/api/detect} API along with an optional camera source identifier.

\subsection{Backend Decoding}
The server removes the data-URL prefix (if present) and decodes the base64 data into raw image bytes, which are passed to the detection module.

\subsection{Hierarchical Detection (Fallback Chain)}
The system uses a multi-level detection strategy:
\begin{itemize}
    \item \textbf{Primary:} AWS Rekognition is used if credentials are valid.
    \item \textbf{Secondary:} If unavailable, a local YOLO model is used.
    \item \textbf{Tertiary:} If both fail, demo mode generates synthetic detections.
\end{itemize}

\subsection{Output Normalization}
All detection results are converted into a unified format containing:
\begin{itemize}
    \item Label
    \item Confidence score (\%)
    \item Detection type
    \item Bounding box (Left, Top, Width, Height)
    \item Display color
    \item Alert flag (is\_alert)
\end{itemize}

\subsection{Threat / Alert Decision}
An alert is triggered only if:
\begin{itemize}
    \item The detected label matches a predefined threat label
    \item The confidence score exceeds the alert threshold
\end{itemize}

\subsection{Logging}
Detection results are stored in the database if logging is enabled or if an alert is triggered. Stored data includes label, confidence, bounding box, timestamp, and source.

\subsection{Alert Dispatch with Cooldown}
For each alert:
\begin{itemize}
    \item A cooldown mechanism prevents repeated alerts
    \item If allowed:
    \begin{itemize}
        \item Snapshot is saved
        \item Event is logged
        \item Notifications are sent via SMS, Telegram, or Email
    \end{itemize}
\end{itemize}

\section{Summary}

This project presented \textbf{EyeQ -- Smart Vision with Intelligence}, a web-based intelligent surveillance platform that combines computer vision, a Flask backend, and multi-channel alerting to support near--real-time monitoring. The system ingests video from common sources such as webcams and RTSP streams, compresses frames in the browser, and transmits them to the server for analysis. A unified detection pipeline prioritizes AWS Rekognition when cloud credentials are available, otherwise uses local YOLO inference, and finally provides a demo mode so the user interface remains operational during configuration or partial failures. Detections are normalized into a consistent representation with bounding boxes and confidence scores, enabling a single client implementation for visualization and analytics.

The engineering contribution is not only the use of a detector, but the construction of a coherent workflow: authenticated access, REST-style APIs for live processing, persistence of detection and alert events, and a dashboard that summarizes activity for operators. This structure separates concerns between capture, inference, policy evaluation, storage, and presentation, which improves maintainability compared with standalone scripts that perform detection without integrated logging or user management.

The implementation couples detection outputs with policy-driven alerting. Configurable detection and alert thresholds, threat-label matching, and per-label cooldown reduce redundant notifications while preserving responsiveness when repeated frames contain the same object class. Events are persisted for auditing and reporting, and operators can review activity through a dashboard with summary statistics and charts, alongside dedicated alert history views. Notification channels including SMS (Twilio), Telegram, and email allow practical deployment in different environments and communication preferences.

Overall, EyeQ demonstrates an end-to-end workflow from frame capture to storage and operator notification, aligning software engineering practices with applied computer vision for campus-scale surveillance use cases. The project also shows that practical surveillance software must address throughput trade-offs (resolution, interval, logging policy) and operational concerns (credentials, cooldown, and channel availability), not only model accuracy in isolation.

\section{Limitations}

The accuracy and reliability of automated surveillance depend on several factors that remain challenging in real deployments. Detection performance is influenced by lighting, occlusion, motion blur, camera resolution, and scene complexity, and no single model guarantees perfect recognition in unconstrained environments. Small or distant objects, reflective surfaces, and crowded scenes can increase missed detections or ambiguous labels. The project’s backends differ in capability: YOLO is constrained by the COCO label set unless retrained or extended, while Rekognition depends on network availability, latency, and cost considerations. Text-related features may require additional system dependencies (such as OCR components), which increases setup complexity on some machines. Consequently, the system should be interpreted as a decision-support tool rather than a fully autonomous security authority.

From an engineering standpoint, the current design targets development and demonstration workflows. Session handling, HTTPS deployment, hardening against abuse, and scalable job processing are not fully addressed in a default local configuration. Continuous detection with database writes can stress low-end hardware unless frame rate, resolution, and logging policies are tuned. Role-based access control is present at a basic level but may require stricter separation of duties for production multi-user deployments, including audit trails for administrative actions and clearer data-retention policies for stored frames and metadata.

Ethical and governance limitations are also important. Surveillance analytics can affect privacy and proportionality if deployed without oversight. The system should be used with transparent institutional rules, minimized data collection relative to the stated security objective, and clear accountability for who can access alerts and historical records. These limitations define the boundary between a functional academic prototype and a hardened enterprise surveillance product, and they motivate the enhancements discussed in the following section.

\section{Future Enhancements}

Several extensions would strengthen EyeQ for broader adoption and more demanding operational settings. A custom-trained detector or fine-tuned model tailored to campus-specific objects and scenes would improve precision and reduce irrelevant alerts compared with generic pretrained classes. Dataset curation, labeling discipline, and periodic evaluation on representative camera feeds would be essential to avoid overfitting to narrow scenarios.

Introducing asynchronous notification queues (for example, Celery or RQ) would decouple inference from message delivery and improve robustness under burst traffic. This is especially useful when multiple alert-worthy detections occur within short intervals, or when external APIs (SMS/email) experience transient failures and require retries without blocking the detection loop.

Replacing periodic polling with WebSocket or server-push updates could reduce latency for dashboards and improve real-time situational awareness for operators monitoring multiple feeds. Similarly, background jobs could precompute hourly summaries and anomaly indicators to reduce client-side workload.

Packaging the stack with Docker and providing environment profiles for PostgreSQL would simplify reproducible deployment beyond developer laptops. Containerization also supports staged rollout: development, staging, and production configurations with separate credentials and logging levels.

Longer-term improvements include a richer device and camera registry for multi-site management, stronger RBAC with audit logs for administrative actions, and deeper analytics such as incident timelines and exportable compliance reports. Integrating edge inference (on-camera or on-gateway processing) could reduce bandwidth and cloud dependency for selected sites. Additional hardening steps would include rate limiting on APIs, structured security logging, automated backups for databases, and monitoring of service health (queue depth, notification failure rates, and detection latency distributions).

"""
Flask Surveillance System — Main Entry Point
Run: python app.py
"""

from datetime import timedelta
from flask import Flask, redirect, send_from_directory, url_for
from flask_login import LoginManager
from models import db, User
import config

login_manager = LoginManager()

def create_app() -> Flask:
    app = Flask(__name__, template_folder="templates", static_folder="static")

    # Config
    app.config["SECRET_KEY"] = config.SECRET_KEY
    app.config["SQLALCHEMY_DATABASE_URI"] = config.DATABASE_URL
    app.config["SQLALCHEMY_TRACK_MODIFICATIONS"] = False

    # Session settings
    app.config["SESSION_PERMANENT"] = True
    app.config["PERMANENT_SESSION_LIFETIME"] = timedelta(hours=24)
    app.config["REMEMBER_COOKIE_DURATION"] = timedelta(days=7)
    app.config["REMEMBER_COOKIE_HTTPONLY"] = True

    # Initialize extensions
    db.init_app(app)
    login_manager.init_app(app)
    login_manager.login_view = "auth.login"

    # Register Blueprints
    from routes.auth import auth_bp
    from routes.dashboard import dashboard_bp
    from routes.camera import camera_bp
    from routes.alerts import alerts_bp
    from routes.detection import detection_bp
    from routes.moderation import moderation_bp

    app.register_blueprint(auth_bp)
    app.register_blueprint(dashboard_bp)
    app.register_blueprint(camera_bp)
    app.register_blueprint(alerts_bp)
    app.register_blueprint(detection_bp)
    app.register_blueprint(moderation_bp)

    @app.route("/")
    def index():
        return redirect(url_for("dashboard.index"))

    with app.app_context():
        db.create_all()

    return app

if __name__ == "__main__":
    app = create_app()
    app.run(debug=config.DEBUG)

# flask_rekognition/models.py

"""Database models — Flask-SQLAlchemy ORM"""

from datetime import datetime
from flask_sqlalchemy import SQLAlchemy
from flask_login import UserMixin
from werkzeug.security import generate_password_hash, check_password_hash

db = SQLAlchemy()


class User(UserMixin, db.Model):
    __tablename__ = "users"

    id            = db.Column(db.Integer, primary_key=True)
    username      = db.Column(db.String(64),  unique=True, nullable=False)
    email         = db.Column(db.String(120), unique=True, nullable=True)
    password_hash = db.Column(db.String(256), nullable=False)
    role          = db.Column(db.String(16),  default="admin")
    created_at    = db.Column(db.DateTime,    default=datetime.utcnow)

    def set_password(self, pw):
        self.password_hash = generate_password_hash(pw)

    def check_password(self, pw):
        return check_password_hash(self.password_hash, pw)

    def is_admin(self):
        return self.role == "admin"


class AlertEvent(db.Model):
    __tablename__ = "alert_events"

    id               = db.Column(db.Integer, primary_key=True)
    object_detected  = db.Column(db.String(128), nullable=False)
    confidence_score = db.Column(db.Float,   nullable=False)
    detection_type   = db.Column(db.String(32),  default="yolo")
    camera_source    = db.Column(db.String(256), default="webcam")
    image_path       = db.Column(db.String(512), nullable=True)
    alert_sent       = db.Column(db.Boolean,     default=False)
    sms_sent         = db.Column(db.Boolean,     default=False)
    timestamp        = db.Column(db.DateTime,    default=datetime.utcnow, index=True)

    def to_dict(self):
        return {
            "id": self.id,
            "object_detected": self.object_detected,
            "confidence_score": round(self.confidence_score, 2),
            "detection_type": self.detection_type,
            "camera_source": self.camera_source,
            "image_path": self.image_path,
            "alert_sent": self.alert_sent,
            "sms_sent": self.sms_sent,
            "timestamp": self.timestamp.isoformat() + "Z",
        }


class DetectionLog(db.Model):
    __tablename__ = "detection_log"

    id             = db.Column(db.Integer, primary_key=True)
    label          = db.Column(db.String(128), nullable=False, index=True)
    confidence     = db.Column(db.Float,   nullable=False)
    detection_type = db.Column(db.String(32),  default="yolo")
    bounding_box   = db.Column(db.String(256), nullable=True)
    camera_source  = db.Column(db.String(256), default="webcam")
    timestamp      = db.Column(db.DateTime,    default=datetime.utcnow, index=True)

    def to_dict(self):
        import json
        return {
            "id": self.id,
            "label": self.label,
            "confidence": round(self.confidence, 2),
            "detection_type": self.detection_type,
            "bounding_box": json.loads(self.bounding_box) if self.bounding_box else None,
            "camera_source": self.camera_source,
            "timestamp": self.timestamp.isoformat() + "Z",
        }
